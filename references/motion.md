# Woolwork motion

All motion expresses one idea: the interface is soft. Springs, settles, and give; never linear easing, never a hard stop, never a continuous effect that decorates rather than acts.

## What motion is not

A continuous pointer-tracked deformation (a dent or tilt that follows the cursor across a surface) was built and cut. It read as an illusion layered on top of the object rather than a property of the object itself, and it is not to be reintroduced under any name. Motion in this system only ever accompanies a real state change: a press, a check, an open, an arrival. Nothing animates just because the pointer is nearby.

## Press feedback on buttons and sewn controls

Never scale a text-bearing element on press. Scaling re-rasterizes text and makes dashed stitches crawl. The kit's pattern is a 1.5px translate settle plus a deepened inset shadow, so a press reads as the patch sinking very slightly into the board and popping back, not as the button shrinking.

## Sew-check strokes

The checkbox (`.sew-check`) is a four-hole button that gets cross-stitched when checked. The X is sewn in two thread strokes tied to the press phases: pointer down draws the first stroke, from one hole to the diagonally opposite hole (about 130ms), pointer up draws the second and the box is checked. Unchecking unpicks in reverse: pointer down removes the second stroke, pointer up removes the first. Each stroke is a pseudo-element anchored at one hole's exact pixel position, rotated to the opposite hole, and grown with `scaleX` from that anchor, so the endpoints always land in the holes regardless of theme or dye. The kit implements the press phase with a `.pre` class set on pointerdown. Keyboard and label activation play both strokes in quick sequence via transition delays. Native checkbox semantics are untouched.

## Place, then stitch (reveal choreography)

Elements with `.sew` appear on the board when scrolled into view with an opacity-only fade of about 0.6 seconds: no translate, no rotate, no transform of any kind, so a revealing element occupies exactly its settled position, outline, and size at every visible frame, and nothing on the page ever appears anywhere other than where it will rest. The physical "sewn on" motion belongs to the thread, not the patch: if the element also has `.stitch`, the running stitch sews on around the edge, appearing segment by segment from the top-left corner over roughly 1.75 seconds, starting just after the fade, and the path it draws along is its final resting path from the first frame.

The reveal observer's root extends far above the viewport, so content the user jumps past without scrolling over it (anchor navigation, an End-key jump, a scrollbar drag) reveals immediately instead of staying invisible until they happen to scroll back across it. Content still below the viewport keeps the normal arrival choreography.

When JavaScript runs, the stitch on a `.stitch` element is built from the actual individual SVG dashes that remain around the card. The kit waits until a sewn card approaches the viewport, then measures it, follows its four hand-cut corner radii at an equal inset, and creates the final dashes before the reveal begins, so cards far below the viewing path carry no unused SVG subtree. Sewing reveals those exact nodes one at a time with a discrete opacity step after the patch reaches full opacity. There is no mask and no replacement path, so every visible animated dash has the same color, weight, edge coverage, and opacity it keeps after settling. A ResizeObserver updates those persistent dash paths when the box changes size, adding or removing only the final dashes when the perimeter needs a different count. The CSS dashed border is the no-JS fallback only, and is hidden the instant `.ww-js` is present on the document. A seam (`.seam.sew`) is the exception: it has no box to trace, so its crease and running stitch wipe on from left to right instead, the stitch trailing a beat behind the crease.

Rules:

- Reveals are visual only: content exists immediately, page height never changes, selection and find-in-page work from the first frame, and the page is fully correct with JavaScript disabled.
- Reveals run once per element per page load and remove their scaffolding afterward. This is intentional: the choreography represents the patch being placed on the board, an arrival, not a loop. A genuine full-page reload always replays it, since nothing persists across navigations. If a host environment restores a page from back-forward cache without re-running scripts, treat that the same as a fresh load rather than leaving elements in their revealed-but-unobserved state.
- Stagger by DOM order comes free from the IntersectionObserver. Do not add artificial delays beyond roughly 0.4s total per viewport. Use the `--sew-delay` custom property for hand-placed staggers within a single group (a grid of cards, a row of chips).
- `.sew` carries no transform at all, hidden, mid-reveal, or settled: the reveal is pure opacity. Never reintroduce a translate or rotate on it. Besides moving the element off its settled geometry, any transform value (even zero) creates a stacking context that traps dropdown panels and tooltips inside the card under later sibling cards.
- Add the one-line `ww-js` snippet to the document head (see `assets/starter.html`) so the pre-reveal hidden state applies before first paint. Without it, a slow first paint can show content, hide it, then reveal it, which reads as a glitch rather than an arrival. Pair it with the starter's `onerror` disarm on the kit script tag so a failed script load cannot leave content hidden.

## Spring vocabulary

- Overlay arrival (toasts, modals): the kit's `--spring` easing, 0.45 to 0.55s. These are top-layer or fixed elements arriving over the page, so their entrance translate never moves page content. In-flow `.sew` reveals are opacity-only instead.
- Fastening (checkbox cross, toggle knob, snap radio): same spring, 0.5s.
- Micro press: 0.08 to 0.13s ease-out down, spring back up.
- Continuous ambient motion is rare and slow (a spinner winding, a basting pulse). At most a small handful of such elements should ever be visible at once, and both stop under `prefers-reduced-motion`.

## Gotchas

- Only transform and opacity animate per frame. Never transition `filter`, `border`, `box-shadow` in a loop (single-shot hover shadow transitions are fine).
- `details.flap` panels must stay absolutely positioned. Converting them to in-flow will reflow the page on open.
- The enhanced select keeps a real native select for form submission and no-JavaScript fallback. Its visible control is one properly sized paper sheet, sized and aligned like the pocket inputs beside it. Its aligned, rounded paper options unfold directly from its own bottom edge with alternating static fold faces and an opaque, transform-only paper backing. The backing prevents nearby copy from showing through between moving folds. Opening carries the concertina motion. Closing hides the panel immediately and resets its folds while hidden, because edge-on reverse folds read as transparent bars over the next control rather than believable paper.
- Popups (`dialog.pinned`) are paper, not felt: a sticky note pinned above the board, closed with the pencil-drawn cross (`.pencil-x`), never a clip, a yarn-x, or a `.btn-patch`. Actions inside use `.btn-ink`. Text inputs use `.ink-line`, never `input.pocket`/`textarea.pocket`. A select inside a popup keeps `select.pocket`, because that class is the kit's enhancement hook, not a felt look: with JS its visible face is the paper choice sheet the kit builds (the native select is hidden), and without JS `select.pocket` itself renders as a paper slip. Nothing visibly felt appears inside the note in either state. The note is capped to the visible viewport and scrolls internally only when browser zoom or a short screen leaves too little room. This keeps every unfolded paper option reachable without unlocking the board behind it. Its backdrop is a flat dim scrim with no `backdrop-filter`. Blurring a live sample of the page is unreliable once the page has other 3D-transformed layers and at non-100% browser zoom, where Chromium and Firefox have both been seen to render the sampled backdrop mirrored, flipped, or tiled. A flat scrim has no such failure mode at any zoom or scroll position and still reads as a dim, physically stable board behind the note. The board behind an open note is scroll-locked (`body:has(dialog.pinned[open])`, reinforced by a kit JS watcher for browsers without `:has()`) via plain `overflow:hidden`, which preserves the exact prior scroll offset on its own (no `position:fixed` scroll-jacking needed). `html{scrollbar-gutter:stable}` is permanent, not conditional, so the viewport width, and every fixed-positioned sibling's offset, never changes when the lock toggles. Mark a dialog's opening button `.pinned-trigger`. Since `showModal()` is always script-invoked, a dialog can never open without JS, so the kit hides `.pinned-trigger` and `.pinned-close` and renders the dialog's own markup in flow as a static preview once `.ww-js` is absent, rather than shipping a dead button.
- Tab faces (`.tabs>button::before/::after`) are flat vector geometry, never a `perspective`/`rotateX` 3D lean. Rasterizing a textured `background-image` through a 3D transform forces the browser to resample it through a perspective matrix, which shows up as visible jagged, stairstepped edges on the slanted side at any zoom other than 100%. A crisp corner radius reads as a hand-cut folder tab just as well and stays anti-aliased at every zoom level.
- Keep toasts and modals outside component containers. The kit appends at most four toasts to one fixed `.toast-tray` on body (so they stack in a column, each dismissable by its yarn-cross or a sideways swipe) and uses the native dialog top layer for modals. A toast pauses its timer while hovered or focused.
- The dropdown flap panel is a slip of paper that slides down out of a slot behind its trigger. The trigger must stay above it (the kit gives `.flap>summary` a higher `z-index`) so the paper reads as emerging from underneath. Do not add `.felt` to the panel.
- Stitched elements are positioned by the kit so their stitch always traces their own box. Overlays (dropdown panels, open flaps) carry explicit z-index above sibling cards. Never wrap an overlay's ancestor in a new stacking context (isolation, transform, filter) or the overlay will paint beneath later siblings.
- Respect the kit's `prefers-reduced-motion` block. Add any new animation inside the same guard discipline.
- One Tier 3 simulation (canvas or WebGPU cloth hero) per viewport at most, feature-detected, paused off-screen. See `frameworks.md` for the WebGPU posture.
